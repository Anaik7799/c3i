1 Pi CLI Interface

Pi is an AI coding assistant with file read, bash execution, structured edit, and write tools. It supports 23 providers, 839 models, session persistence, extensions, skills, and themes.

Full CLI Help Output

pi - AI coding assistant with read, bash, edit, write tools

Usage:
  pi [options] [@files...] [messages...]

Commands:
  pi install <source> [-l]     Install extension source and add to settings
  pi remove <source> [-l]      Remove extension source from settings
  pi uninstall <source> [-l]   Alias for remove
  pi update [source]           Update installed extensions (skips pinned sources)
  pi list                      List installed extensions from settings
  pi config                    Open TUI to enable/disable package resources
  pi <command> --help          Show help for install/remove/uninstall/update/list

Options:
  --provider <name>              Provider name (default: google)
  --model <pattern>              Model pattern or ID (supports "provider/id" and optional ":<thinking>")
  --api-key <key>                API key (defaults to env vars)
  --system-prompt <text>         System prompt (default: coding assistant prompt)
  --append-system-prompt <text>  Append text or file contents to the system prompt (can be used multiple times)
  --mode <mode>                  Output mode: text (default), json, or rpc
  --print, -p                    Non-interactive mode: process prompt and exit
  --continue, -c                 Continue previous session
  --resume, -r                   Select a session to resume
  --session <path>               Use specific session file
  --fork <path>                  Fork specific session file or partial UUID into a new session
  --session-dir <dir>            Directory for session storage and lookup
  --no-session                   Don't save session (ephemeral)
  --models <patterns>            Comma-separated model patterns for Ctrl+P cycling
                                 Supports globs (anthropic/*, *sonnet*) and fuzzy matching
  --no-tools                     Disable all built-in tools
  --tools <tools>                Comma-separated list of tools to enable (default: read,bash,edit,write)
                                 Available: read, bash, edit, write, grep, find, ls
  --thinking <level>             Set thinking level: off, minimal, low, medium, high, xhigh
  --extension, -e <path>         Load an extension file (can be used multiple times)
  --no-extensions, -ne           Disable extension discovery (explicit -e paths still work)
  --skill <path>                 Load a skill file or directory (can be used multiple times)
  --no-skills, -ns               Disable skills discovery and loading
  --prompt-template <path>       Load a prompt template file or directory (can be used multiple times)
  --no-prompt-templates, -np     Disable prompt template discovery and loading
  --theme <path>                 Load a theme file or directory (can be used multiple times)
  --no-themes                    Disable theme discovery and loading
  --no-context-files, -nc        Disable AGENTS.md and CLAUDE.md discovery and loading
  --export <file>                Export session file to HTML and exit
  --list-models [search]         List available models (with optional fuzzy search)
  --verbose                      Force verbose startup (overrides quietStartup setting)
  --offline                      Disable startup network operations (same as PI_OFFLINE=1)
  --help, -h                     Show this help
  --version, -v                  Show version number

Extensions can register additional flags (e.g., --plan from plan-mode extension).

3 User Journey 1: Interactive Coding Session

The primary Pi workflow: launch an interactive session, select a model, write prompts, and let the agent use its tools to read, edit, and execute code.

$ pi
-->
Select model (Ctrl+P)
-->
Write prompt
-->
Agent uses tools
-->
Results displayed

Step 1: Launch Pi

User launches pi from the terminal. Pi loads AGENTS.md and CLAUDE.md context files, discovers extensions and skills, and presents the interactive prompt.

$ pi
# or with a specific model:
$ pi --model anthropic/claude-sonnet-4-20250514
# or with thinking enabled:
$ pi --model sonnet:high "Analyze the architecture"

Step 2: Model Selection

Press Ctrl+P to cycle through available models at runtime. Pi supports glob patterns and fuzzy matching. Models can be pre-filtered with --models:

# Cycle between specific models
$ pi --models "claude-sonnet,gemini-2.5-pro,gpt-4o"

# Cycle all models from a provider
$ pi --models "anthropic/*"

Step 3: Agent Tool Execution

The agent autonomously decides which tools to use based on the prompt. Default tools are read, bash, edit, write. Read-only mode available with --tools read,grep,find,ls.

User: "Refactor the database module to use connection pooling"

Agent actions:
  1. read  -- Read current db.gleam source
  2. grep  -- Search for connection patterns across codebase
  3. edit  -- Apply structural refactoring with find/replace
  4. bash  -- Run `gleam build` to verify compilation
  5. bash  -- Run `gleam test` to verify tests pass
  6. write -- Create new pool_config.gleam if needed

Step 4: Session Management

Sessions are automatically saved as JSONL files. Resume any session with --continue, --resume (picker), or --fork (branch from existing session).

# Continue the last session
$ pi --continue "What were we working on?"

# Pick a session to resume
$ pi --resume

# Fork a session into a new branch
$ pi --fork <session-uuid>

4 User Journey 2: Pi + C3I Zenoh Bridge

Pi agent events are bridged into the C3I mesh via pi_agent.gleam. Every Pi action (tool call, message, error) is published as an OTel span on the Zenoh backplane, making Pi a first-class citizen of the SIL-6 Biomorphic Mesh.

-->
pi_agent.gleam bridge
-->
Zenoh publish
-->
C3I Dashboard

Bridge Architecture

The Gleam bridge module at bridge/pi_agent.gleam translates Pi's JSONL session events into C3I-native AG-UI 32-event protocol messages. These are published to Zenoh topics:

Zenoh Topic Mapping:
  indrajaal/pi/agent/tool_call    -- Pi tool invocations (read, edit, bash, write)
  indrajaal/pi/agent/message      -- Pi assistant messages
  indrajaal/pi/agent/error        -- Pi errors and failures
  indrajaal/pi/agent/session      -- Session lifecycle (start, end, fork)
  indrajaal/otel/spans/pi/{op}    -- OTel spans for distributed tracing

Dashboard Integration

The C3I dashboard at port 4100 subscribes to indrajaal/pi/agent/** topics via WebSocket. Pi tool calls appear in real-time on the Agents page alongside Cortex OODA loop events. The weather bar reflects Pi activity in the cognitive (L5) layer health score.

Transport Protocol

Pi extension sends JSON-RPC request:
  Topic: indrajaal/mcp/req/knowledge_search/{request_id}
  Body:  { "jsonrpc": "2.0", "method": "knowledge_search", "params": { "query": "OODA loop" } }

  Topic: indrajaal/mcp/res/{request_id}
  Body:  { "jsonrpc": "2.0", "result": { "holons": [...], "count": 12 } }

Total round-trip: ~4ms (Zenoh pub/sub + Smriti FTS5 lookup)

6 User Journey 4: Session Persistence

Pi sessions are persisted as JSONL files. The C3I bridge module pi_session.gleam mirrors session state into Smriti.db, making Pi sessions recoverable across server restarts and available for Zettelkasten recall.

Pi session (JSONL)
-->
pi_session.gleam
-->
Smriti.db (SQLite)
-->
Recoverable across restarts

Persistence Flow

1. Pi writes session events to ~/.pi/agent/sessions/{path}/session.jsonl
2. pi_session.gleam watches the JSONL file (or receives events via Zenoh)
3. Each event is parsed and stored in Smriti.db:
     - pi_sessions table: session metadata (model, provider, start time, token usage)
     - pi_messages table: conversation turns (role, content, tool calls, timestamps)
     - pi_tool_calls table: tool invocations (name, args, result, duration)
4. On server restart, sessions are recoverable from Smriti.db
5. Pi session summaries are indexed in the Zettelkasten FTS5 for recall

Session Recovery Commands

# Resume last Pi session via Pi CLI
$ pi --continue

# Search past Pi sessions via C3I Zettelkasten
$ sa-plan-daemon knowledge-search "pi session refactoring db module"

# Export a Pi session to HTML for sharing
$ pi --export ~/.pi/agent/sessions/abc123/session.jsonl report.html

7 User Journey 5: Hedged Inference

When Pi is configured to use the C3I inference cascade via pi_provider.gleam, requests are hedged across multiple model tiers. The fastest successful response wins, providing sub-second latency for simple queries and deep analysis for complex ones.

Pi request
-->
pi_provider.gleam
-->
C3I 6-tier cascade
-->
Fastest response wins

6-Tier Hedged Inference Cascade

TierModelLatencyCostTransport
1Gemini Direct (flash-lite)~900msFreeHTTPS
2OpenRouter (flash-preview)~1.1s$0.000009HTTPS
3Ollama gemma4 (port 11435)~4sFreeHTTP
4Ollama gemma3 (port 11434)~10sFreeHTTP
5RETE-UL rule engine (52 GRL)<1msFreeIn-process
6Static acknowledgment<1msFreeIn-process

Tiers 1+2 fire simultaneously via tokio::join!. First success wins. 4 independent circuit breakers (3 failures leads to 60s cooldown). No-blackhole guarantee: 7 mechanisms ensure every request gets a response.

How It Works

1. Pi sends inference request via pi_provider.gleam extension
2. pi_provider.gleam publishes to Zenoh: indrajaal/mcp/req/inference/{id}
3. sa-plan-daemon cortex.rs receives the request
4. Hedged dispatch: Tier 1 (Gemini) || Tier 2 (OpenRouter) fire in parallel
5. First successful response is returned via Zenoh: indrajaal/mcp/res/{id}
6. If both fail: cascade to Tier 3 (local Ollama gemma4)
7. If all cloud/local models fail: Tier 5 (RETE-UL rules) provides deterministic answer
8. Response delivered back to Pi with PipelineTracer footer:
   Pipeline: recv(0ms) > class(1ms) > ack(2ms) > infer(1200ms) > delivered(1400ms)
