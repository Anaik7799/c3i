3. Implementation Approach

3.1 Flock mutex (R1)

// Before
"command": "cd /home/an/dev/ver/c3i && ... ingest-docs ... && ... fy27-zettelkasten import .. ..."

// After
"command": "flock -n /tmp/c3i-stop-hook.lock -c '...same chain...' || echo '{\"systemMessage\":\"Stop-hook ingest already running, skipped (flock).\"}'"

Non-blocking (-n): if lock held, exits immediately with friendly JSON. The lock file persists; flock holds an exclusive advisory lock on it. Multiple Claude sessions ending simultaneously: first wins, others no-op.

3.2 c3i.slice (R6)

[Unit]
Description=C3I service slice — CPU-bounded to ≤ 70% of total host capacity

[Slice]
CPUQuota=700%       # 70% × 10 cores; kernel CFS bandwidth controller enforces
CPUWeight=200       # competitive weight when slice not saturated
MemoryHigh=32G      # soft throttle: triggers reclaim
MemoryMax=38G       # hard kill: cgroup-OOM, doesn't propagate to host
IOWeight=200
TasksMax=4096

Per-unit drop-ins use the same primitives at finer granularity. Hierarchical accounting: child sums can exceed slice; CFS arbitrates fairly via CPUWeight when slice cap is hit.

6.2 Runtime behaviour change

  • Resource fairness: when c3i.slice consumes > 70 % of CPU, kernel CFS bandwidth controller throttles tasks within the slice. Other user processes (Claude Code, browser, IDE) are untouched.
  • Memory pressure: at 32 G slice usage, kernel triggers reclaim. At 38 G, cgroup-OOM kills the largest task in the slice. Host stays responsive.
  • Hook idempotency: Stop hook now safe to fire arbitrarily often; first wins, rest skip.
  • Observability: any operator (or remote tailnet peer) sees state at :8090/health/services.json with at most 30 s lag.
  • Port safety: a misconfigured second service binding 4100 will be refused start by systemd via Conflicts=.
  • Periodic re-eval: ops-status fires every minute, slo-guard every minute, history-compactor every 10 min — system noticing drift continuously, not only at boot.

7. Operations Runbook

7.1 Triage: dashboard slow

  1. curl http://localhost:8090/health/services.json | jq '.slice_cpu_nsec, .slice_memory_bytes'
  2. If CPU near 7e9 ns/sec → slice throttle. Check cat .../c3i.slice/cpu.stat for nr_throttled.
  3. If memory > 32 G → reclaim active. Check memory.pressure.
  4. If neither: check upstream (gemini API, openrouter circuit breakers).

7.2 Triage: hook fired but ZK not updated

  1. Check ls -la /tmp/c3i-stop-hook.lock; expected if another ingest active.
  2. journalctl --user -t bash --since '5 min ago' for "Stop-hook ingest already running, skipped".
  3. pgrep -af 'sa-plan-daemon ingest-docs'; expected at most one.

7.3 Tuning the cap

# Reduce to 50% (5 cores)
sed -i 's/CPUQuota=700%/CPUQuota=500%/' ~/.config/systemd/user/c3i.slice
systemctl --user daemon-reload
# No restart needed — kernel applies cap to running cgroup live
