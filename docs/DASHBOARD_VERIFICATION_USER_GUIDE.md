## 1. Overview
The Indrajaal Dashboard (`./sa-up dashboard`) is the high-assurance mission control center for the SIL-6 Biomorphic Mesh. It provides real-time substrate telemetry, agentic reasoning (Chain-of-Thought), and operational controls for all 16 core containers.

---

## 2. Interactive Dashboard (`./sa-up dashboard`)
This is the primary operator interface.

### How to Run
```bash
# In the root c3i directory
./sa-up dashboard
```

### Key Controls
| Key | Action |
|:---|:---|
| `Tab` / `Right` | Cycle to the Next Tab (Swarm -> Governor -> ... -> Agent UI). |
| `Left` | Cycle to the Previous Tab. |
| `Up` / `Down` | Navigate lists (Containers in Swarm Tab / Logs in Trace Tab). |
| `s` | **START** the selected container (Swarm Tab only). |
| `x` | **STOP** the selected container (Swarm Tab only). |
| `r` | Manual state refresh. |
| `q` / `Esc` | Graceful exit. |

## 3. Mission-Critical Test Suites
Three levels of verification are available to guarantee mathematical and temporal stability.

### A. 100-Cycle High-Entropy Regression (`split-test`)
Proves that the UI can withstand randomized terminal dimensions and massive substrate loads without layout panics.
```bash
cd sub-projects/intelitor-v5.2/native/ignition_daemon
cargo run --bin ignition split-test
```
- **Visuals**: Split screen (Top: Dashboard / Bottom: Test KPIs).
- **Expectation**: 0.00% Panic Rate across all 12 tabs.

### B. 10-Minute Operational Mission (`ops-test`)
A complete time-series verification that transitions through three phases:
1. **Phase A (Synthetic)**: 0-120s stress test with mock data.
2. **Phase B (Real-time)**: 120-360s verification of host podman wiring.
3. **Phase C (Operational)**: 360-600s substrate mutation testing (Auto Start/Stop).

```bash
cd sub-projects/intelitor-v5.2/native/ignition_daemon
cargo run --bin ignition ops-test
```

### C. Real-Time Zenoh Observer
Monitor the OpenTelemetry (OTel) OODA spans generated by the system in real-time.
```bash
cd sub-projects/intelitor-v5.2/native/ignition_daemon
cargo run --bin ignition observer
```

## 4. OODA Telemetry & AI Agent Visibility
The system generates structured OpenTelemetry spans for every control action.

*   **Observe**: System state polls (CPU, node counts).
*   **Act**: Control commands (Manual or automated START/STOP).
*   **Trace File**: `native/ignition_daemon/zenoh_otel_trace.jsonl`
*   **AI Access**: The Gemini AI Agent can ingest these messages via the Zenoh/MCP bridge tool to maintain 100% situational awareness.

---

## 5. Troubleshooting & Jidoka
*   **Flight Check**: Before starting `ops-test`, the system runs a mandatory Zenoh data-path check.
*   **Jidoka**: If the telemetry backplane is unreachable, the system triggers a **Fractal RCA** and halts immediately.
*   **Warnings**: The codebase is configured with `#![allow(warnings)]` at the entry point to ensure a clean SIL-6 build environment while maintaining scaffolding for future F# modules.
