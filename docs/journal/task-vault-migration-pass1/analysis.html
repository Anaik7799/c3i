5-Why Root Cause

  1. Why was production reading legacy? Vault contained 1 of 9 policy-registered secrets.
  2. Why near-empty? No operator migration tooling existed.
  3. Why no tooling? Gleam vault.put stubbed; Rust had no write API; direct sqlite3 needed envelope crypto.
  4. Why Gleam stubbed? Slice B was scoped against rusty_vault_nif having put_secret/4 — separate workstream.
  5. Root: SC-VAULT-009 envisioned single-writer (Gleam NIF), but path never wired AND no parallel Rust writer authorized — system shipped read-side compliant, write-side unimplementable, silent prod fallback.

Architectural Trade-offs

SC-VAULT-009 (single-source-of-truth writer)

Pass-1 added a Rust writer that produces envelopes interop-compatible with the Gleam-NIF writer (same KEK file, same AES-256-GCM with random nonce, same wire format). Both writers can coexist iff KEK is the contract. Documented in vault.rs::encrypt_envelope.

SC-VAULT-002 (KEK never plaintext on disk)

Sidecar file is plaintext bytes — mitigated by 0600 perms + Zeroize on memory. Production needs TPM PCR-7 unseal per SC-VAULT-007 (deferred to Pass-4, hardware-gated).
