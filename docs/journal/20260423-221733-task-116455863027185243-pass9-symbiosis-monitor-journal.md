## 1. Problem framed via Fractal TPS

Previous passes shipped features but operators kept **re-searching** the same
system facts every session — DB paths, binary locations, service units, URL
templates. That is **Muda** (waste). It also meant there was no single source
of truth against which an agent could *verify symbiosis* between ZK (smriti),
Pi, and the active Gleam/Elixir agents.

We needed:

1. A canonical, cache-able snapshot of system facts every agent boots against
2. A **closed-loop OODA monitor** that continuously measures ZK fill + token
   cost + cache health, so drift is surfaced before it accumulates
3. A **live dashboard** a customer can open at the LE-signed Tailscale URL
4. **Zero manual work** when a session starts — no "where is smriti?"

- `databases.smriti_stale` / `smriti_corrupt` / `smriti_stub` → listed **explicitly** as "do not touch"
- `binaries.sa_plan_daemon_live` / `sa_plan_daemon_source` + `rebuild_cmd`

- `journal_conventions` + `url_template` (`customer` / `internal` / `localhost`)
- `parent_chain` — p5 through p9
- `gleam_modules` — KMS / Breaker / Httpx / Nif / per-feature modules
- `nif_functions_scripts_nif` — the 17 active NIF exports
- `agent_ops_checklist` — 7-item boot-time protocol

This **eliminates the re-search** loop for every future agent session. It is
the Jidoka "knowledge board" for the shopfloor.

### 2.2 `p9_symbiosis_monitor` — OODA loop Gleam module

`sub-projects/scripts-gleam/src/scripts/pass9/p9_symbiosis_monitor.gleam`
(≈ 350 LOC, zero new deps, reuses `common/kms` + `common/nif`).

One tick = Observe → Orient → Decide → Act, completes in **3–20 ms**:

- **Observe**: 10 SQL reads against live smriti (holons, edges, embeddings,
  session_metrics, session_turn_spans, pi_sessions, prompt_cache, semantic_cache,
  pipeline_stage_metrics, rolling token/cost totals), plus a 1-row projection of
  the most recent `session_metrics` (agent, provider, model, tokens, zk_recalls,
  cache_hit_ratio).
- **Orient**: derived ratios — `embed_coverage = embeddings / holons`,
  `tokens_per_session`, `cost_per_session`, `cache_hit` and `cache_entries`.
- **Decide**: four alarm rules — `EMBED_LOW` (< 50 %), `TURN_SPANS_EMPTY`
  (Pi turns not traced), `CACHE_HIT_LOW` (< 30 % on ≥ 1 000-tok session),
  `COST_HIGH` (> $ 5 / session), `PI_NO_TURN_TRACE` (pi sessions w/o spans).
- **Act**: atomic write of JSON snapshot + Zenoh span publish on
  `indrajaal/l5/ooda/symbiosis`.

### 2.3 `c3i-symbiosis-monitor.service` — systemd unit

`~/.config/systemd/user/c3i-symbiosis-monitor.service`, enabled + running.
5 000 ms tick, auto-restart, logs to `data/monitor/monitor.log`. Wires PATH
to the nix-store `gleam` so the service starts clean at boot.

### 2.4 Dashboard HTML

`docs/journal/monitor/symbiosis.html` served at
`https://vm-1.tail55d152.ts.net/c3i/task-id/any/monitor/symbiosis.html`
through Tailscale's LE cert — **no warnings, works from any customer browser.**

- 6-column card grid: holons / embed coverage / sessions / tokens / cost / cache-hit
- Color-codes card on thresholds (good / warn / bad)
- `last_session` table with agent / provider / model / in-out tokens / cost

- Raw JSON panel (collapsed, debug-friendly)
- Auto-polls every 2 s; pulse dot indicates snapshot age
- Override via `?src=…&ms=…` query params

## 5. SRE & symbiosis effect

The monitor is the **L5 cognitive-cortex OODA process** of the fractal. Before
p9 we had per-pass snapshots; after p9 the system self-observes every 5 s.

- `.c3i-context.json` gives every agent a deterministic bootstrap — agent
  diff across Claude / Pi / Gemini / Codex converges
- JSON API lets any downstream tool subscribe without re-reading SQLite
