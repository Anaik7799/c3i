## 9. Architectural Observations

1. **The Rust FP ecosystem is post-fp-core.** The 2024-2026 maturation of `winnow` (1.0), `rpds` (1.x), `nutype` (0.7), `tailcall` (2.x) means a curated 12-library stack is now feasible without authoring custom typeclass libraries.
2. **C3I's existing constraint families are FP-shaped without using FP language.** SC-WIRE = type-system discipline. SC-VALUE-GUARD = refinement discipline. SC-VAULT = sealed-state discipline. CPIG = composite-gate discipline. Adding SC-FP-RUST formalizes the substrate that already exists.
3. **The right adoption order privileges vault.** nutype on `rusty_vault_nif` is the highest-leverage Pass-2 move because it lifts SC-VAULT-001..025 from runtime to type level for the smallest surface (~5 newtypes) with the strongest payoff (vault is L0 constitutional).
4. **rayon and tokio are mutually-exclusive across surfaces.** Mixing them is the highest-RPN failure mode (RPN 210). The discipline must be: rayon-offline-only, tokio-online-only, never both in the same module.
5. **kani fills a real gap.** C3I has TLA+ for state machines and Agda for type-level proofs but no Rust-source bounded model checking. kani is the missing leg of the verification stool.
6. **No new libraries should be authored.** The optics/effects/STM/free-monad gaps in the Rust ecosystem are real, but a SIL-6 system gains more from leaning on existing constraint families than from authoring generic FP libraries that would dilute the safety story.
7. **FP discipline is a Lyapunov problem, not a one-shot install.** FP_TOTAL must increase monotonically across passes (λ ≥ 0). A pass that adds libraries without proving KPI movement is a pass that didn't ship.

### RCA: dual-winnow-version build failure

**Symptom**: First attempt used `winnow::combinator::delimited` for the markdown parser. `cargo test --lib fp_demo` returned E0283 — the compiler couldn't disambiguate which `delimited` to call.

**5-Why root cause**:
1. **L1 Symptom**: `delimited` ambiguous trait resolution on line 220
2. **L2 Surface**: Two `winnow` versions in dependency graph (0.7.15 + 1.0.1)
3. **L3 System**: Pass-4 added `winnow = "0.7"` to planning_daemon/Cargo.toml; `mistralrs::EmbeddingModelBuilder` (used in `embedding.rs`) transitively pulls `winnow 1.0` via `toml_parser`
4. **L4 Process**: Pass-4 fp_demo only used `winnow` combinators (`alt`, `take`, `take_while`) that didn't trigger the ambiguity at the trait-bound level. `delimited` does, because its trait bounds are stricter
5. **L5 Cultural**: When two viable versions of a parser library coexist via different dependency paths, neither cargo nor rustc can pick one — the application code must explicitly opt out of one path or use full type ascription

**Resolution**: Rewrote `parse_markdown_task_line` to use stdlib `str::find` / `str::rfind` / `as_bytes()` matching — zero parser combinator usage, zero version conflict. Pass-4 `parse_task_line` retains winnow combinators (the demo intent). Linter-added `winnow::error::ErrMode<winnow::error::ContextError>` annotations were reverted to plain `|e|` (winnow 0.7 needs `ContextError`, not `ErrMode<ContextError>` in `.map_err`).

**Honest scope statement** [zk-3346fc607a1ef9e6]: this finding should constrain Pass-8+ winnow usage on planning_daemon. Either (a) drop the direct `winnow` dep entirely and use stdlib parsing, or (b) bump direct dep to `winnow = "1.0"` and accept that Pass-4 parser code needs migration. Operator-review-grade choice.

### Design rationale: additive parallel API

The legacy `add_task(&str, &str)` is preserved because c3i_nif (the Erlang term decoder) accepts raw strings from BEAM and cannot pass refined types across the FFI boundary without a custom NIF resource type. Migrating that boundary is itself a separate operator-review-grade refactor.

`add_task_refined(&str, &Priority)` is the new path for callers that have already lifted their priority through `Priority::try_new` (e.g., the SCRIPT-level CLI, future Lustre→Wisp REST handlers, internal Rust caller graph). It skips the runtime `validate_priority` check because the type system has already proven the value is canonical — saving ~50ns per call and replacing a dynamic dispatch with a static one.

### What Pass-9 does NOT do

- No c3i_nif boundary migration (Erlang term encoder still uses `&str`)
- No Status counterpart (`add_task_refined_status`-style API for `update_task_status` is straightforward Pass-10 work)
- No criterion bench measuring the type-vs-runtime-validate latency delta (small, but quantifiable)
- No breaking change to existing callers — `add_task` keeps working unchanged
