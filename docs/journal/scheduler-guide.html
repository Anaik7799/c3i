1. The big picture

sa-plan-daemon ships two execution engines sharing one SQLite file (Smriti.db). Both are orthogonal to planning tasks (sa-plan add/update/status) — your 3,077-row todo list.

EngineBorrowed fromExecutesLifetimeUse when
OBAN Elixir Oban Short, idempotent jobs seconds–minutes Cron-driven background work, fire-and-forget tasks
TEMPORAL Temporal.io Long, stateful sagas minutes–days Multi-step flows that must survive restarts, accept signals/queries
PLANNING (C3I-native) Human intent / todo state permanent What the operator wants done; not directly executable

5. The five commands in detail

schedule-list — cron definitions

./sa-plan schedule-list
  • Backing table: workflow_schedules
  • Auto-seeds defaults if empty (per WF-5).
  • Companions: schedule-add, schedule-pause, schedule-trigger, schedule-configure.

job-list — concrete jobs

./sa-plan job-list                                  # last 20 (default)
./sa-plan job-list --queue embed --state executing  # filtered
./sa-plan job-list --json                           # machine-readable
./sa-plan job-list --limit 100
  • Backing table: oban_jobs (JobRow per oban.rs:249).
  • States: scheduled · available · executing · retryable · completed · discarded · cancelled.
  • Companions: job-enqueue, job-cancel, job-retry, job-snooze, job-prune, job-lifeline.

queue-list — resource pools

./sa-plan queue-list
  • Backing tables: oban_queues joined with aggregate over oban_jobs.
  • Default queues (from oban.rs:164): default, critical, maintenance, ingest, delivery — each max_concurrency=4.
  • Companions: queue-pause, queue-set-concurrency.

workflow-executions — long-running sagas

./sa-plan workflow-executions                              # last 20
./sa-plan workflow-executions --workflow-type ingest_corpus
./sa-plan workflow-executions --state running --json
  • Backing table: workflow_executions (PK: workflow_id, run_id).
  • Sagas are durable: a daemon restart resumes from the last workflow_event.
  • Companions: workflow-start, workflow-signal, workflow-query, workflow-cancel, workflow-terminate, workflow-describe, workflow-continue-as-new.

workflow-list — event history

./sa-plan workflow-list
  • Backing table: workflow_events (per [zk-3d6c2ca4c011d463]).
  • Append-only; this is the durable history feed and the ground truth for replay.

7. How to call them

From the shell

# Inspectors (read-only)
./sa-plan schedule-list
./sa-plan job-list --queue ingest --state available --json --limit 50
./sa-plan queue-list
./sa-plan workflow-executions --state running
./sa-plan workflow-list

# Mutations
./sa-plan schedule-add  --cron "0 */6 * * *" --worker embed_refresh
./sa-plan job-enqueue   --worker embed_refresh --queue ingest --priority 5
./sa-plan job-cancel    4711
./sa-plan job-retry     4711
./sa-plan job-snooze    4711 --delay-seconds 300
./sa-plan workflow-start ingest_corpus --input '{"path":"docs/"}'
./sa-plan workflow-signal abcd-1234 pause
./sa-plan workflow-query  abcd-1234 progress
./sa-plan workflow-cancel abcd-1234

From the daemon (programmatic)

// Rust — oban.rs::enqueue_opts
let id = oban::enqueue_opts(EnqueueOptions {
    queue: "ingest",
    worker: "ingest_docs",
    args_json: r#"{"path":"docs/"}"#,
    meta_json: None,
    tags: &["nightly", "ingest"],
    priority: 5,
    max_attempts: 3,
    scheduled_at: None,
    unique_key: Some("ingest_docs/docs/"),  // dedupe within window
    unique_period_secs: 3600,
    unique_states: None,
    workflow_id: None,
})?;

Live observer over Zenoh

./sa-plan sched-observe                       # live envelopes on indrajaal/l4/sched/**

9. How to execute tasks — three patterns

Pattern A — fire-and-forget job (single shot)

Best for: idempotent one-off work, no inter-step state.

./sa-plan job-enqueue --worker embed_refresh --queue ingest --priority 5
./sa-plan job-list --limit 5             # confirm enqueue
# scheduler tick will pick it up; concurrency capped by oban_queues.max_concurrency
./sa-plan job-list --state completed     # done

Pattern B — recurring background work (cron)

Best for: maintenance jobs (embed_refresh, zk_maintain, prune_jobs, sa_plan_sync).

./sa-plan schedule-add --cron "0 */6 * * *" --worker embed_refresh \
                       --queue ingest --priority 5 --max-attempts 3
./sa-plan schedule-list                  # appears
./sa-plan schedule-trigger <id>          # fire now (debug)
./sa-plan schedule-pause   <id>          # operator override

Pattern C — durable saga (multi-step)

Best for: ingest pipelines, multi-stage builds, anything that must survive restarts and accept signals.

./sa-plan workflow-start ingest_corpus --input '{"path":"docs/"}'
# returns workflow_id (e.g. abcd-1234)

./sa-plan workflow-executions             # state=running
./sa-plan job-list --workflow-id abcd-1234   # child jobs the saga spawned

./sa-plan workflow-signal abcd-1234 pause
./sa-plan workflow-query  abcd-1234 progress      # synchronous question

./sa-plan workflow-list                   # full event history
./sa-plan workflow-describe abcd-1234     # state + replay tree

./sa-plan workflow-cancel abcd-1234       # cooperative
./sa-plan workflow-terminate abcd-1234    # hard stop (last resort)

Composed example — ingest 200 docs as a saga of jobs

// Pseudo-saga in workers.rs — workflow_executions row drives this
1. workflow-start ingest_corpus → INSERT workflow_executions (state=running)
2. saga decides: split docs/ into 10 chunks
3. for each chunk:
     enqueue oban_jobs row (worker=ingest_docs, workflow_id=abcd-1234)
4. wait_for_all_children() — saga subscribes to job state changes
5. on ALL completed → emit WorkflowCompleted, state=completed
6. on ANY discarded ≥ retry-budget → emit WorkflowFailed, state=failed

10. Operator recipes

Daemon idle when it shouldn't be

./sa-plan schedule-list                # any schedules paused?
./sa-plan queue-list                   # any queues paused or concurrency=0?
./sa-plan job-list --state available   # is anything queued at all?
./sa-plan job-list --state retryable   # backoff in progress?

Stuck job

./sa-plan job-list --state executing   # find stale id N
./sa-plan job-snooze N --delay-seconds 0
# or bulk
./sa-plan job-lifeline                 # reset all jobs stuck > N seconds

Hung workflow

./sa-plan workflow-executions --state running
./sa-plan workflow-describe <wf-id>
./sa-plan workflow-cancel    <wf-id>     # cooperative
./sa-plan workflow-terminate <wf-id>     # hard-stop (last resort)

Bypass cron

./sa-plan schedule-trigger <schedule-id>
./sa-plan job-list --limit 5             # see the just-enqueued job

12. Relation to planning tasks

These five commands do not enumerate the 1,852 pending planning tasks. Planning tasks (planning_status table, governed by SC-TODO-001) are a separate domain — intent records mutated only via sa-plan add/update.

The single bridge: a sa_plan_sync job (or ooda_recommend) can mutate planning tasks. So a workflow may schedule jobs that advance the todo list. But job-list / workflow-list never show planning tasks directly. Use sa-plan dashboard (TUI) or sa-plan serve (web :4200) for the rich task view.

DomainTablesCLI surface
PLANNING planning_status, planning_history add, update, status, recommend, dashboard, serve
SCHEDULER + WORKFLOW oban_jobs, oban_queues, workflow_schedules, workflow_executions, workflow_events, workflow_signals, workflow_queries, workflow_timers 5 list commands + their mutators (schedule-add, job-enqueue, workflow-start, …)
