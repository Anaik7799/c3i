# NixOS Architect Skill

This skill provides advanced patterns for managing NixOS systems at scale.

## 1. Modular Configuration
NixOS systems should be split into logical components.

### Structure Recommendation
- `hosts/`: Machine-specific hardware and boot settings.
- `modules/`: Reusable profiles (e.g., `desktop.nix`, `server.nix`, `k3s.nix`).
- `configuration.nix`: The entry point that imports the necessary modules.

### Rule of Separation
- **Hardware**: Always keep `hardware-configuration.nix` separate (autogenerated).
- **Software**: Define declarative lists of `environment.systemPackages`.

## 2. Secrets Management
**Critical**: Never put raw secrets in the Nix store.

- **Best Practice**: Use `sops-nix` or `agenix` for encrypted secrets.
- **Legacy Pattern**: Use `deployment.keys` or `services.<name>.passwordFile` pointing to a local path like `/var/lib/secrets/`.

## 3. System Lifecycle
- **`nixos-rebuild switch`**: Standard update.
- **`nixos-rebuild test`**: Non-persistent testing (reverts on reboot).
- **Rollbacks**: Use `nix-env --rollback` or select a previous generation in the bootloader.

## 4. Modern Flakes
Always use Flakes for hermetic reproducibility.
```nix
{
  inputs.nixpkgs.url = "github:nixos/nixpkgs/nixos-unstable";
  outputs = { self, nixpkgs }: {
    nixosConfigurations.nas-1 = nixpkgs.lib.nixosSystem {
      modules = [ ./configuration.nix ];
    };
  };
}
```
