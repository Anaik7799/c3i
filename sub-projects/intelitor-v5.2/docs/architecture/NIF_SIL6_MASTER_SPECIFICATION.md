## 4. 7-Level Fractal Architecture

The NIF subsystem is organized into 7 fractal levels of detail:

### Level 1: Cellular (Atomic Code)
*   **Component**: Rust Source (`native/zenoh_nif/src/lib.rs`).
*   **Logic**: Zero-cost abstractions over the `zenoh` C/Rust API.
*   **Safety**: `catch_unwind` ensures Rust panics never crash the BEAM.

### Level 2: Tissue (Elixir Binding)
*   **Component**: `Indrajaal.Native.Zenoh` module.
*   **Logic**: `check_nif_loaded/0` and fallback stubs.
*   **Config**: `skip_compilation?: not @cargo_available` logic for resiliency.

### Level 3: Organ (Process Integration)
*   **Component**: `Zenoh.Publisher` GenServer.
*   **Logic**: Manages `ResourceArc` lifecycles.
*   **Safety**: Uses **Dirty Schedulers** (`DirtyCpu`) for all blocking ops (SC-NIF-L2).

### Level 4: System (Container Runtime)
*   **Component**: `indrajaal-app` Container.
*   **Design**: **Trojan Horse** image with injected `pkgs.cargo` and `pkgs.rustc`.
*   **Config**: `containers/sopv51-elixir-app.nix`.

### Level 5: Organism (Node)
*   **Component**: Podman Node.
*   **Behavior**: Rootless execution (`User 0:0` mapped to host user).
*   **Control**: `podman-compose-fractal-mesh.yml`.

### Level 6: Colony (Fractal Mesh)
*   **Component**: Zenoh Control Plane.
*   **Data Flow**: High-frequency telemetry (<1ms) over TCP/UDP.
*   **Control Flow**: Pub/Sub based orchestration.

### Level 7: Ecosystem (Global Federation)
*   **Component**: CEPAF / Digital Twin.
*   **Behavior**: OODA Loop integration (Observe NIF stats, Decide scaling, Act on mesh).

## 5. SIL-6 Compliance

### 5.1 Data Flow & Quadplex Telemetry
To meet SIL-6 observability standards, NIFs emit data to four simultaneous channels:
1.  **Console**: `IO.puts` (Immediate operator feedback).
2.  **File**: Durable logs in `logs/` (Audit trail).
3.  **Prometheus**: Metrics `indrajaal_nif_calls_total` (Quantitative analysis).
4.  **Zenoh**: Native pub/sub (Real-time control loop).

### 5.2 Digital Twin Architecture
The F# CEPAF Cockpit maintains a **Digital Twin** of the NIF state.
*   **Shadow**: Every NIF session has a shadow representation in the Twin.
*   **Synchronization**: State changes (Connect/Disconnect) are published to Zenoh `indrajaal/nif/lifecycle`.
*   **Drift Detection**: The Twin alerts if the reported state diverges from observed telemetry.

### 5.3 Transaction Behavior
NIF updates follow an **Atomic Transaction** model:
1.  **Prepare**: Compile new `.so` artifact.
2.  **Verify**: Run L1-L5 tests.
3.  **Commit**: Atomic filesystem move to `priv/native/`.
4.  **Rollback**: If load fails, revert to previous `.so`.

### 7.1 STAMP Safety Constraints
*   **SC-NIF-001**: NIFs MUST NOT block the scheduler (>1ms ops must be Dirty).
*   **SC-NIF-002**: Resources MUST be cleaned up via `Drop` trait implementation.
*   **SC-NIF-003**: All NIF errors MUST be trapped and propagated as tuples.

### 7.2 FMEA (Failure Mode & Effects Analysis)
*   **Risk**: Toolchain missing in container. **Mitigation**: Trojan Horse Build.

*   **Risk**: Segfault. **Mitigation**: Rust memory safety + catch_unwind.
