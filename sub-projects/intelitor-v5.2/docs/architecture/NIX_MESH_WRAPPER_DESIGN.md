# DESIGN NOTE: Nix-Native Tailscale Wrapper Architecture

**Date:** 2025-12-19
**Status:** PROPOSED
**Classification:** INFRASTRUCTURE / BUILD SYSTEM
**Component:** NixOS Containers (Infrastructure Layer)
**Reference:** Task 22.1.2 (Infrastructure Mesh)

## 1. The Challenge
Our infrastructure containers (`postgres`, `redis`, `nginx`) are built using `pkgs.dockerTools.buildImage`. Unlike Dockerfiles, which are imperative (Run this, Copy that), Nix builds are declarative and functional.
*   **Constraint 1:** We cannot simply "inject" a shell script into an existing image without rebuilding the layers.
*   **Constraint 2:** These containers are often minimal, lacking tools like `ip`, `iptables`, or even `sh` in some cases (though ours include bash).
*   **Constraint 3:** Nix store paths are read-only. We cannot modify the entrypoint script in place.

## 2. The Solution: Functional Wrapper Pattern
We will define a reusable Nix function, `wrapWithTailscale`, that acts as a higher-order function. It takes an existing derivation (the original entrypoint) and wraps it inside a new shell script that initializes the mesh.

### 2.1 The Interface
```nix
# containers/lib/tailscale.nix
{ pkgs }:
{
  # Returns a derivation containing the 'tailscaled' and 'tailscale' binaries
  # and a 'wrap' function.
  
  # Function: wrap
  # Args:
  #   originalEntrypoint: The path or string of the command to run after Tailscale starts
  #   name: Name for the wrapper script
  wrap = originalEntrypoint: pkgs.writeShellScriptBin "entrypoint-with-tailscale" ''
    ... script content ...
  '';
}
```

### 2.2 The Wrapper Logic (Runtime)
The wrapper script generated by Nix will perform the following sequence:
1.  **Environment Setup:** Source defaults for `TS_STATE_DIR` (/var/lib/tailscale).
2.  **Tun Device Creation:** Check for `/dev/net/tun`. If missing (rootless), fallback to userspace networking mode.
3.  **State Directory Prep:** Ensure `/var/lib/tailscale` exists and is writable (critical for Nix containers which might have read-only rootfs; we rely on the volume mount).
4.  **Daemon Start:** Launch `${pkgs.tailscale}/bin/tailscaled`.
5.  **Authentication:** Run `tailscale up --authkey=$TS_AUTHKEY`.
6.  **Handoff:** `exec` the original entrypoint.

## 3. Implementation Plan

### 3.1 Create the Library
New file: `containers/lib/tailscale.nix`.
*   **Inputs:** `pkgs`.
*   **Outputs:** `package` (the tailscale derivation), `wrap` (the function).

### 3.2 Refactor Database Container
Modify `containers/indrajaal-timescaledb-demo.nix`:
1.  Import the library: `ts = import ./lib/tailscale.nix { inherit pkgs; };`
2.  Add `ts.package` to `contents` (so binaries exist in the image).
3.  Wrap the command: `Cmd = [ "${ts.wrap entrypoint}/bin/entrypoint-with-tailscale" ];`

### 3.3 Refactor Orchestrator
Update `podman-compose.yml` to support the new needs of these containers:
*   **Volumes:** Create `tailscale_db_state`, `tailscale_redis_state`.
*   **Environment:** Inject `TS_AUTHKEY` and `TS_HOSTNAME`.

## 4. Security & Isolation
*   **Identity:** Each infrastructure component gets its own discrete identity on the Tailnet (e.g., `db-prod`, `redis-cache`).
*   **ACLs:** We can use Tailscale ACLs to strictly limit access.
    *   *Rule:* `tag:app` can access `tag:db` on port 5433.
    *   *Rule:* `tag:dev` (human) can access `tag:db` on port 5433 (for debugging).
    *   *Rule:* `tag:db` cannot initiate connections to `tag:app`.

## 5. Potential Friction Points
*   **Startup Latency:** Adding Tailscale negotiation adds ~2-5 seconds to container startup. Health checks must account for this (increase `start_period`).
*   **Volume Permissions:** The `tailscaled` process runs as root (or rootless-root). The DB runs as `postgres`. We must ensure the volume mounts don't conflict or block permissions. The wrapper runs as root (PID 1) before switching users, so it can handle `mkdir`.

## 6. Verification
Success is defined as:
1.  `podman-compose up -d postgres` starts successfully.
2.  `tailscale ping timescaledb-localhost` works from the host machine.
3.  The App container can connect via `DATABASE_URL=postgres://...@timescaledb-localhost.tailnet.ts.net:5433/...`
