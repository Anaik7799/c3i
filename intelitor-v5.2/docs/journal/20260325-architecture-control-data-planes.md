# Architectural Overview: Control Plane vs. Data Plane in Indrajaal SIL-6
**Date**: 2026-03-25
**Author**: Cybernetic Architect (Gemini)
**Status**: ACTIVE

## Executive Summary
The Indrajaal SIL-6 Biomorphic architecture achieves extreme resilience, performance, and sovereignty by strictly bifurcating its operations into a **Control Plane** (deciding *what* to do and *how* to route it) and a **Data Plane** (doing the actual heavy lifting and storage). This prevents heavy computational tasks from starving the system's ability to orchestrate and heal itself.

---

## 1. The Control Plane (The Nervous System & Brain)
The Control Plane is responsible for orchestration, health monitoring, routing, policy enforcement, and cognitive decision-making. It operates on low-latency, high-priority pathways.

### Core Elements
*   **Zenoh Router Mesh (`zenoh-router-1, 2, 3`)**: The physical backbone of the control plane. It provides a zero-copy, ultra-low-latency message bus using a 2oo3 (2-out-of-3) quorum. It routes telemetry, commands, and inference requests without the overhead of HTTP.
*   **F# Cortex (`indrajaal-cortex`)**: The infrastructure orchestrator. It manages the SIL-6 boot sequence, evaluates FPPS consensus for health checks, and runs the authoritative F# Planning System.
*   **CEPAF Bridge (`cepaf-bridge`)**: The actuator for the Cortex. It holds the Podman socket, allowing the control plane to physically spawn, kill, or restart containers in the data plane based on health metrics.
*   **The Guardian (Elixir Safety Kernel)**: A deterministic rules engine sitting between the AI and the system. It acts as the ultimate control-plane gatekeeper, vetoing any unsafe "YOLO" mutations before they reach the data plane.
*   **Synapse / Hybrid Router (Elixir)**: The routing intelligence. It looks at an incoming task, determines its complexity, and routes it either to local fast-heuristics (Nx) or the heavy compute plane (Mojo/MAX).

### Key Functionality
*   **Homeostasis & Autonomy**: Monitors the health of all nodes and automatically restarts failed components (via CEPAF).
*   **Traffic Shaping**: Determines where tasks should run (e.g., offloading to FLAME runners).
*   **Security & Policy**: Enforces the Founder's Directive and STAMP safety constraints.

---

## 2. The Data Plane (The Muscle & Memory)
The Data Plane is responsible for executing business logic, processing high-throughput AI inference, serving user requests, and durably storing state. It operates on high-bandwidth, compute-intensive pathways.

### Core Elements
*   **Application HA Cluster (`indrajaal-ex-app-1, 2, 3`)**: The primary execution engine for user requests. Built on Elixir/Phoenix, it serves the Prajna Cockpit UI, handles API endpoints, and executes fast, deterministic logic (the "Reflex Core" using `Nx` and `Bumblebee`).
*   **Mojo MAX Compute Node (`indrajaal-mojo`)**: The heavy AI inference engine. When the Control Plane dictates that a task requires deep reasoning (SLMs like Llama 3), the data is routed here. It uses Modular's MAX engine for maximum hardware utilization (CPU/GPU) without blocking the BEAM nodes.
*   **FLAME Satellites (`indrajaal-ml-runner-1, 2`)**: Elastic compute muscle. The application cluster can dynamically spin these up to offload specific, heavy data-processing tasks.
*   **SMRITI State Storage (`data/holons/`)**: The system's immutable "Soul." SQLite (for fast WAL state) and DuckDB (for append-only historical analytics) store the actual configuration, memory, and evolutionary lineage of the system.
*   **Business Persistence (`indrajaal-db-prod`)**: PostgreSQL 17 + TimescaleDB. Strictly used for transactional application data and time-series metrics, kept entirely separate from the system's own holonic state.
*   **Observability Stack (`indrajaal-obs-prod`)**: The data sink for telemetry. SigNoz, ClickHouse, and Prometheus ingest massive streams of logs and metrics generated by the data plane, which the control plane then queries to make decisions.

### Key Functionality
*   **Execution**: Running the actual code, rendering UIs, and generating tokens (LLM inference).
*   **Throughput**: Moving large volumes of data (e.g., video streams, bulk database queries).
*   **Persistence**: Safely writing state to disk.

---

## Synthesis: How They Interact
1.  A user asks the **Data Plane** (Phoenix App) to solve a complex problem.
2.  The **Control Plane** (Synapse Router) intercepts the request via the Zenoh bus, determines it needs high intelligence, and routes the prompt to the **Data Plane's** compute node (Mojo/MAX).
3.  The Mojo node processes the data and sends the result back.
4.  The **Control Plane** (Guardian) intercepts the result, verifies it doesn't violate safety constraints, and then allows the **Data Plane** (Phoenix) to render it to the user.
5.  Simultaneously, if the Mojo node uses too much CPU, the **Control Plane** (Sentinel/Cortex) detects the metric spike via the Observability stack and throttles the node or spins up a FLAME satellite to share the load.

# Comprehensive Plan: Robust NixOS Observability Container & F# Lifecycle Supervisor

**Created**: 20260325-1200 CEST
**Status**: ACTIVE
**Framework**: SOPv5.11 + TPS (Jidoka + 5-Level RCA) + SC-CNT-009

## Executive Summary
The `indrajaal-obs-prod` container is currently unhealthy because it relies on a broken NixOS skeleton image (`sleep infinity`). To achieve SIL-6 biomorphic integration and comply with SC-CNT-009 (NixOS-only environments), this plan completely rebuilds the container using a strict, declarative `flake.nix` (leveraging `dockerTools.buildLayeredImage`). 

Crucially, an F# Agent (`.NET 10.0`) will be engineered to act as the primary container supervisor. Running under a lightweight init (`tini`) to handle zombie reaping, the F# Agent will boot before any observability services. It will establish native Zenoh mesh connectivity, expose Model Context Protocol (MCP) endpoints, and autonomously manage the lifecycles of Prometheus, Grafana, ClickHouse, and OpenTelemetry. 

## Detailed Architectural Analysis for Robustness

### 1. PID 1 & Signal Handling
- **Challenge**: If an F# application runs directly as PID 1, it must handle OS signal propagation (SIGTERM/SIGINT) and reap zombie child processes (Prometheus, Grafana, etc.). Failure to do so leads to resource exhaustion and ungraceful shutdowns.
- **Robust Solution**: Use `tini` (provided via Nixpkgs) as the absolute PID 1. `tini` will spawn the F# Agent, guaranteeing proper zombie reaping and signal forwarding. The F# Agent will catch `SIGTERM` and orchestrate a graceful shutdown of its managed child services.

### 2. The F# Supervisor (Cepaf.ObsSupervisor)
- **Role**: Replaces `start-obs.sh` with a fully typed, stateful `MailboxProcessor` actor system.
- **Responsibilities**:
  - Spawn processes: `prometheus`, `grafana-server`, `clickhouse-server`, `otelcol`.
  - Monitor process exits and implement exponential backoff restarts.
  - Expose a unified `/health` endpoint for the `podman` healthcheck, aggregating the status of all managed services.
- **Runtime**: `.NET 10.0` (mandated by SC-NET-001).

### 3. Mesh Connectivity (Zenoh + MCP)
- **Zenoh FFI**: The Nix build must compile or include `libzenoh_ffi.so`. The F# agent will bind to this library to publish health telemetry (`indrajaal/obs/health`) and listen for control plane commands (e.g., `indrajaal/obs/cmd/restart`).
- **MCP Server**: The agent will embed an MCP server exposing tools (`obs_status`, `obs_restart_service`, `obs_read_service_log`) allowing Gemini/Claude direct introspective capabilities into the observability stack without needing bash access.

### 4. Pure NixOS Build Pipeline
- **Mechanism**: A dedicated `containers/obs/flake.nix`.
- **Contents**: It will define a derivation that compiles the F# Agent and packages it alongside Prometheus, Grafana, ClickHouse, OTEL, and `tini` into a layered OCI image. No Dockerfiles will be used.
